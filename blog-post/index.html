Sequential Fine-Tuning and Catastrophic Forgetting in LLMs

Author: Sahan Randika Kalutarage

Institution: The University of Texas at San Antonio (UTSA)

Topic: LLM Post-Training, Alignment, and Imitation Learning

1. Methodology

This project investigates the complex dynamics of post-training and sequential fine-tuning in Large Language Models (LLMs). Specifically, we aim to quantify the trade-offs between acquiring highly structured, rigid capabilities (JSON generation) and retaining general, fluid conversational alignment. To achieve this, we engineered a two-stage training pipeline utilizing imitation learning, accompanied by an ablation study on epoch scaling.

1.1 Model Selection

Student Model: A highly capable but compact open-weight base model (microsoft/Phi-3.5-mini-instruct) was selected as the student. Smaller parameter models are ideal for post-training experiments as they exhibit more pronounced shifts in behavior during fine-tuning compared to their larger counterparts, making phenomena like catastrophic forgetting easier to measure within academic compute constraints. During training, the base model is dynamically quantized to 4-bit (nf4) precision using bitsandbytes (with double quantization and float16 compute type) to drastically reduce VRAM footprints.

Teacher Model: We utilized a frontier-level model (llama-3.3-70b-instruct-awq) deployed locally to generate synthetic training data. These large-scale models possess robust internal representations of complex schema structures, making them ideal teachers for distilling strict reasoning frameworks into smaller student architectures.

1.2 Data Sourcing and Imitation Learning Pipeline

The training corpus was divided into two distinct datasets to support the sequential training stages:

  • Stage 1 (General Alignment): We utilized the open-source Stanford Alpaca dataset (tatsu-lab/alpaca), which contains 52,000 instruction-following records. A 90/10 train-validation split was utilized. It covers a broad taxonomy of tasks and serves as the academic standard for transforming a base completion model into a conversational assistant.
  • Stage 2 (Structured Output): To teach the model strict structural compliance, we constructed a synthetic JSON Instruct dataset using an Imitation Learning Pipeline:
    • The teacher model (llama-3.3-70b-instruct-awq) received prompts defining five unique sub-tasks: extraction, json-repair, label-classification, schema-constrained generation, and tool-call formatting.
    • Parametric strictness: The teacher model was generationally constrained using an extremely low temperature (0.1) and enforced system prompts mandating raw string output without markdown blocks.
    • Automated Validation & Retry: The generation pipeline programmatically verified the output syntax. If the teacher hallucinated invalid JSON or injected conversational fluff, the pipeline initiated up to 3 automated retries per sample to guarantee a pristine structural dataset of maximum quality for student distillation.

1.3 Training Design, Ablation Setup, and UTSA HPC Setup

Training was executed sequentially leveraging Parameter-Efficient Fine-Tuning (PEFT) to ensure stable convergence without over-saturating the base model weights.

Hyperparameters (Stages 1 & 2):

  • LoRA Configuration: Rank (r=32), Alpha (64), Dropout (0.05), Target Modules (all-linear), Task Type (CAUSAL_LM).
  • Optimization Strategy: Learning rate of 1e-4 using a cosine scheduler with a 0.3 warmup ratio.
  • Batching & Regularization: Per-device batch size of 4 with 8 gradient accumulation steps (yielding a larger effective batch size). Max gradient norm clipped at 0.3, weight decay at 0.01. Half-precision (fp16) and gradient_checkpointing were enabled for maximal memory efficiency on sequences up to 1024 tokens.

Sequential Pathway:

  • Checkpoint 0: The untuned base model (Phi-3.5-mini-instruct).
  • Checkpoint 1 (Stage 1): Base model fine-tuned exclusively on the Alpaca dataset.
  • Checkpoint 2 (Stage 2): Checkpoint 1 fine-tuned exclusively on the synthetic JSON Instruct dataset.

Ablation Study Design:
To understand how quickly catastrophic forgetting occurs, we conducted an ablation study during Stage 2. Custom arguments (num_train_epochs) were passed to the SLURM batch scripts. We trained independent adapters for 1, 2, and 3 epochs to observe how quickly JSON syntax mastery scales against the deterioration of core conversational fluency.

HPC Configuration:
All computational workloads—including training, evaluation, ablation, and text generation—were executed on the UTSA High-Performance Computing (HPC) cluster. Jobs relied upon SLURM workload manager, allocating 1 compute node on the gpu1v100 partition, ensuring deterministic execution via dedicated V100 GPU acceleration.

1.4 Evaluation Protocol

Evaluation requires moving beyond traditional string-matching. A multi-modal hybrid pipeline was adopted:

Automated Text Metrics:
ROUGE (1, 2, L) and BERTScore (configured with roberta-large) were employed via Hugging Face's evaluate framework to quantify semantic alignment against conversational reference distributions.

Automated Structural Metrics (Custom Engine):
A localized Python assessment engine parsed string outputs to measure:

  • JSON Validity Rate: Binary checks resolving parsing errors.
  • Schema Compliance: Ratio of outputs yielding identically matched Dictionary/Key objects against a reference constraint.
  • Exact Match: Absolute fidelity against ground truth extraction patterns.
  • Field-level F1: Calculated using precision and recall aggregations across deeply nested Key:Value pairings.

LLM-as-a-Judge API:
gpt-4o-mini was utilized via OpenAI's API as an impartial scoring engine for randomized pairwise A/B matchups between checkpoints.

  • Over 200 Alpaca prompts and 100 JSON prompts were assessed.
  • An environment-constrained system prompt penalized the model for hallucination or formatting deviations, utilizing temperature=0.1 and response_format={"type": "json_object"}.
  • Outputs were scored strictly across five dimensions: Instruction Following, Correctness, Clarity, Completeness, and Conciseness on a unified 1-5 scalar system, with randomized response injection orders (A vs B) to mathematically mitigate positional evaluation biases.

2.2 Alpaca Evaluation (Self-Instruct Protocol)

Stage 1 effectively aligned the base model, transforming it from a raw text-completion engine into a conversational assistant. The untuned Base model exhibited a strong tendency to generate overly verbose completions, often hallucinating continued dialogue or formatting tags, which resulted in an average response length of 162.4 tokens.

Stage 1 training aggressively pruned this behavior. The average token length dropped by over 70% to just 45.5 tokens per response. This increase in precision drove a massive gain in automated metrics: ROUGE-1 improved from 0.2744 to 0.4972, and BERTScore F1 rose from 0.8663 to 0.9139. Consequently, in pairwise judge evaluations, Checkpoint 1 significantly outperformed the Base model, achieving a 56.5% win rate (113 wins, 10 ties out of 200 matchups). The judge consistently cited Checkpoint 1's ability to directly address the prompt without tangential rambling as the primary reason for its victories.

2.3 JSON Structured Output Evaluation

Stage 2 was designed to instill strict structural discipline, and it succeeded comprehensively. To understand the improvement, we must look at the Common Error Taxonomy of the earlier models. At Checkpoint 0, the model achieved only a 31% valid parsing rate. An analysis of the failed outputs revealed that the model primarily struggled with Conversational Padding—it would prefix the JSON with helpful but breaking phrases like "Here is the JSON format you requested:". It also suffered from structural truncation, failing to close deeply nested brackets.

By Checkpoint 2, the model had internalized the syntax rules perfectly, achieving a 100% JSON validity rate. It learned to drop all conversational padding, outputting raw, parsable JSON arrays and objects. The semantic mapping of the data within the JSON was also highly accurate, resulting in a BERTScore F1 of 0.9937 and a field-level F1 of 0.8197. Checkpoint 2 defeated Checkpoint 1 in pairwise JSON task evaluations (17 wins to 12, with 71 ties). The high volume of ties indicates that Stage 1 provided a strong baseline, but Stage 2 eliminated all remaining edge-case schema failures.

2.4 Forgetting Analysis

Key Finding: Catastrophic forgetting occurred following Stage 2, primarily as a degradation of stylistic alignment (specifically conciseness) rather than a loss of factual recall or semantic understanding.

The core objective of this research was to measure the "Alignment Tax" incurred when teaching a generalized model a highly rigid task. When evaluated on general Alpaca prompts, Checkpoint 2 regressed significantly compared to Checkpoint 1. It achieved a win rate of only 21.0% (42 wins for Checkpoint 2, 63 for Checkpoint 1, and 95 ties). Automated text metrics mirrored this human-aligned preference drop, with ROUGE-L falling from 0.4335 back down to 0.4197.

To pinpoint the exact cognitive mechanism of this forgetting, we conducted a dimensional analysis based on the LLM judge's sub-scores:

Evaluation Dimension Stage 1 Avg Score Stage 2 Avg Score Absolute Change
Instruction Following 4.71 4.77 +0.06
Correctness 4.80 4.83 +0.03
Completeness 4.50 4.75 +0.25
Conciseness 4.42 4.21 -0.21

This dimensional breakdown is highly illuminating. The model did not forget facts or how to follow instructions; in fact, its Correctness and Completeness scores increased slightly. Because the model was trained exclusively on exhaustive, highly detailed JSON extraction data in Stage 2, it generalized that "exhaustive" behavior to casual conversation. It forgot how to be brief, leading to a sharp drop in conciseness and an overall lower win rate.

3. Analysis & Discussion

3.1 Qualitative Output Comparison & Failure Cases

A qualitative review of the raw outputs provides striking context to the numerical data, demonstrating how the "persona" of the model shifted drastically at each checkpoint.

  • Checkpoint 0 (Untuned Base): The model acted as a raw text-continuation engine. It lacked a conversational persona, often hallucinating additional prompts, leaking template tags, or trailing off into unrelated paragraphs.
  • Checkpoint 1 (Stage 1): The model successfully adopted a helpful, conversational persona. Outputs were direct, neatly formatted into standard paragraphs or simple bullet points, and stopped generating text as soon as the prompt was answered.
  • Checkpoint 2 (Stage 2): The model adopted an ultra-rigid, data-extraction persona. While it was highly accurate, it over-engineered simple answers, applying structured headers and exhaustive lists to basic conversational questions.

An analysis of judge evaluations contextualizes the failure mechanisms at Checkpoint 2. Because Stage 2 training requires structured data extraction, the model over-indexed on this heuristic during general prompts.

Failure Case 1 (Prompt: alpaca_eval_1 - Election Summary):
The prompt requested a summary of a US Presidential Election. Checkpoint 1 generated a standard, concise paragraph. Checkpoint 2 generated a highly structured, verbose breakdown categorizing specific roles like social media and targeted advertising. The evaluator penalized Checkpoint 2, citing that Checkpoint 1 was more concise while still summarizing the core information effectively.

Failure Case 2 (Prompt: alpaca_eval_2 - Economic Prediction):
When asked to predict stock market trends based on rising unemployment, Checkpoint 1 provided a brief, accurate forecast. Checkpoint 2 ignored the mandate for brevity and attempted to generate an exhaustive, multi-variable economic report. The judge again preferred Checkpoint 1 due to its directness.

3.2 Discussion: Stylistic vs. Factual Forgetting

Catastrophic forgetting is typically conceptualized as the destructive overwriting of previously learned weights, leading to a loss of core knowledge or vocabulary. Our results propose a highly nuanced view: Stylistic Forgetting vs. Factual Retention.

In this experiment, Checkpoint 2 did not forget how to answer questions or process facts. This is evidenced by the +0.25 increase in its "Completeness" score and a +0.03 increase in "Correctness." The regression was entirely stylistic; the model misapplied the formatting heuristics of JSON extraction tasks—which inherently demand exhaustive detail—to open-ended conversational instructions that demand brevity.

3.3 Implications for Sequential Fine-Tuning

These results illustrate the severe risks of purely sequential fine-tuning on narrow downstream tasks. When a generalized assistant is sequentially tuned on a rigid task without regularization, it incurs an "Alignment Tax." Two primary strategies can mitigate this stylistic degradation in practical LLM deployment applications:

  1. Early Stopping: The ablation study demonstrates that structural syntax is learned much faster than conversational alignment is unlearned. Terminating the Stage 2 training at Epoch 1 achieves full formatting compliance (100% JSON validity) while minimizing the conversational penalty.
  2. Data Mixing (Experience Replay): Rather than utilizing purely sequential training, Stage 2 should incorporate a mixed data curriculum. By combining the JSON instructions with a 10% to 20% subset of the Stage 1 Alpaca data, the optimizer is forced to anchor the weights associated with general conversational alignment while simultaneously learning the new structured task.

4. Prompt Engineering Evolution

Developing a robust end-to-end pipeline required rigorous prompt engineering on two fronts: the generation of the synthetic training data (via the teacher model) and the refinement of the automated evaluation pipeline (via the LLM-as-a-Judge).

4.1 Imitation Learning Data Generation (Teacher Prompts)

To construct the JSON Instruct dataset, we utilized Claude as the teacher model to generate highly diverse, complex JSON structures. The prompts were engineered to cover five primary task categories. To ensure the student model learned robust generalization rather than memorizing a single format, each primary category was systematically divided into five distinct subcategories:

  • JSON Extraction & Schema Constrained Tasks: Both of these primary categories utilized the same five structural variations to teach the model how to map natural language to specific shapes:
    • Flat Object: Standard key-value pairs (e.g., product attributes, person profiles).
    • Nested Object: Hierarchical data (e.g., an address object inside a person object).
    • Array of Objects: Lists of uniform items (e.g., multiple transaction events or attendees).
    • Mixed Flat + Array: Complex documents (e.g., an invoice with flat metadata and an array of line items).
    • Nullable Fields: Incomplete data extraction (e.g., a clinical note where missing fields must map to null).
  • JSON Repair: Prompts designed to teach syntax correction and parser compliance:
    • Unquoted Keys/Values: Fixing raw JavaScript-style objects missing double quotes.
    • Trailing Commas: Removing illegal commas before closing brackets or braces.
    • Truncated Structure: Closing mid-generation cut-offs (missing braces/brackets).
    • Incorrect Delimiters: Replacing Python-style syntax (single quotes, equal signs) with JSON standards.
    • Escaping Errors: Properly escaping internal double quotes, backslashes, or line breaks.
  • Exact-Label Classification: Prompts constraining the LLM to output specific, predefined categorical labels:
    • Political Stance: Classifying ideological leaning ('LIBERAL', 'CONSERVATIVE', 'CENTRIST').
    • Claim Type: Categorizing epistemological nature ('FACTUAL_CLAIM', 'SUBJECTIVE_OPINION', 'SPECULATION').
    • Support Routing: Determining customer intent ('BILLING', 'TECHNICAL_SUPPORT', 'SALES', 'CANCELLATION').
    • Urgency Triage: Assessing issue severity ('LOW', 'MEDIUM', 'HIGH', 'CRITICAL').
    • Content Moderation: Evaluating intent and safety ('SAFE', 'SPAM', 'TOXIC', 'PHISHING').
  • Tool-Calling: Prompts simulating agentic service calls:
    • Find Flights: Extracting broad search intent (origin, destination, date).
    • Book Flights: Transactional extraction including precise times and passenger counts.
    • Get Weather: Utility calls extracting location and resolving temperature units.
    • Get Product: E-commerce extraction requiring mixed data types (string IDs, integer quantities, float prices).
    • Schedule Meeting: Calendar management parsing 24-hour times, dates, and arrays of email addresses.

Dataset Sizing: For each of the five primary task types, we generated 70 training prompts and 20 testing prompts. This resulted in a strictly balanced and diverse dataset comprising 350 training instances and 100 evaluation instances.

4.2 LLM-as-a-Judge Refinement

Evaluating the resulting models required refining the LLM-as-a-Judge prompt to address inherent evaluation biases observed during early testing.

Evaluation Bias Mitigation: Initial testing revealed that the gpt-4o-mini evaluator exhibited verbosity bias. It frequently assigned high scores to the Base model simply because it produced lengthy text, ignoring critical failure modes like context hallucination or leaked template tags (e.g., ### Instruction:).

Prompt Revision: The judge prompt was revised to include an explicit CRITICAL PENALTIES section. The evaluator was instructed to penalize models heavily for self-prompting, repetitive loops, and unnecessary verbosity. This adjustment aligned the automated scoring more closely with expected human evaluation standards, correctly identifying the verbose outputs of both the Base model and Checkpoint 2 as sub-optimal for general conversational tasks.

Proper response should be concise, short, accurate, and directly address the user's prompt without unnecessary fluff.

- Self-Prompting: Leaking template tags like `### Instruction:` or continuing to generate fake inputs.
- Degenerate Loops: Repeating the same phrases or list structures endlessly.
