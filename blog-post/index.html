Sequential Fine-Tuning and Catastrophic Forgetting in LLMs

Author: Sahan Randika Kalutarage

Institution: The University of Texas at San Antonio (UTSA)

Topic: LLM Post-Training, Alignment, and Imitation Learning

1. Methodology

This project investigates the complex dynamics of post-training and sequential fine-tuning in Large Language Models (LLMs). Specifically, we aim to quantify the trade-offs between acquiring highly structured, rigid capabilities (JSON generation) and retaining general, fluid conversational alignment. To achieve this, we engineered a two-stage training pipeline utilizing imitation learning, accompanied by an ablation study on epoch scaling.

1.1 Model Selection

Student Model: A highly capable but compact open-weight base model (meta-llama/Llama-3.2-3B) was selected as the student. Smaller parameter models are ideal for post-training experiments as they exhibit more pronounced shifts in behavior during fine-tuning compared to their larger counterparts, making phenomena like catastrophic forgetting easier to measure within academic compute constraints. During training, the base model is dynamically quantized to 4-bit (nf4) precision using bitsandbytes (with double quantization and float16 compute type) to drastically reduce VRAM footprints.

Teacher Model: We utilized a frontier-level model (llama-3.3-70b-instruct-awq) deployed locally to generate synthetic training data. These large-scale models possess robust internal representations of complex schema structures, making them ideal teachers for distilling strict reasoning frameworks into smaller student architectures.

1.2 Data Sourcing and Imitation Learning Pipeline

The training corpus was divided into two distinct datasets to support the sequential training stages:

  • Stage 1 (General Alignment): We utilized the open-source Stanford Alpaca dataset (tatsu-lab/alpaca), which contains 52,000 instruction-following records. A 90/10 train-validation split was utilized. It covers a broad taxonomy of tasks and serves as the academic standard for transforming a base completion model into a conversational assistant.
  • Stage 2 (Structured Output): To teach the model strict structural compliance, we constructed a synthetic JSON Instruct dataset using an Imitation Learning Pipeline:
    • The teacher model (llama-3.3-70b-instruct-awq) received prompts defining five unique sub-tasks: extraction, json-repair, label-classification, schema-constrained generation, and tool-call formatting.
    • Parametric strictness: The teacher model was generationally constrained using an extremely low temperature (0.1) and enforced system prompts mandating raw string output without markdown blocks.
    • Automated Validation & Retry: The generation pipeline programmatically verified the output syntax. If the teacher hallucinated invalid JSON or injected conversational fluff, the pipeline initiated up to 3 automated retries per sample to guarantee a pristine structural dataset of maximum quality for student distillation.

1.3 Training Design, Ablation Setup, and UTSA HPC Setup

Training was executed sequentially leveraging Parameter-Efficient Fine-Tuning (PEFT) to ensure stable convergence without over-saturating the base model weights.

Hyperparameters (Stages 1 & 2):

  • LoRA Configuration: Rank (r=32), Alpha (64), Dropout (0.05), Target Modules (all-linear), Task Type (CAUSAL_LM).
  • Optimization Strategy: Learning rate of 1e-4 using a cosine scheduler with a 0.3 warmup ratio.
  • Batching & Regularization: Per-device batch size of 4 with 8 gradient accumulation steps (yielding a larger effective batch size). Max gradient norm clipped at 0.3, weight decay at 0.01. Half-precision (fp16) and gradient_checkpointing were enabled for maximal memory efficiency on sequences up to 1024 tokens.

Sequential Pathway:

  • Checkpoint 0: The untuned base model (Llama-3.2-3B).
  • Checkpoint 1 (Stage 1): Base model fine-tuned exclusively on the Alpaca dataset.
  • Checkpoint 2 (Stage 2): Checkpoint 1 fine-tuned exclusively on the synthetic JSON Instruct dataset.

Ablation Study Design:
To understand how quickly catastrophic forgetting occurs, we conducted an ablation study during Stage 2. Custom arguments (num_train_epochs) were passed to the SLURM batch scripts. We trained independent adapters for 1, 2, and 3 epochs to observe how quickly JSON syntax mastery scales against the deterioration or retention of core conversational fluency.

HPC Configuration:
All computational workloads—including training, evaluation, ablation, and text generation—were executed on the UTSA High-Performance Computing (HPC) cluster. Jobs relied upon SLURM workload manager, allocating 1 compute node on the gpu1v100 partition, ensuring deterministic execution via dedicated V100 GPU acceleration.

1.4 Evaluation Protocol

Evaluation requires moving beyond traditional string-matching. A multi-modal hybrid pipeline was adopted:

Automated Text Metrics:
ROUGE (1, 2, L) and BERTScore (configured with roberta-large) were employed via Hugging Face's evaluate framework to quantify semantic alignment against conversational reference distributions.

Automated Structural Metrics (Custom Engine):
A localized Python assessment engine parsed string outputs to measure:

  • JSON Validity Rate: Binary checks resolving parsing errors.
  • Schema Compliance: Ratio of outputs yielding identically matched Dictionary/Key objects against a reference constraint.
  • Exact Match: Absolute fidelity against ground truth extraction patterns.
  • Field-level F1: Calculated using precision and recall aggregations across deeply nested Key:Value pairings.

LLM-as-a-Judge API:
We utilized a frontier-level judge API (llama-3.3-70b-instruct-awq) to run randomized pairwise A/B matchups between checkpoints. Two distinct prompt templates were strictly engineered to mitigate verbosity bias: an Alpaca judge prompt assessing naturalness and instruction-following, and a separate JSON judge prompt ruthlessly enforcing schema compliance without conversational padding.

2.2 Alpaca Evaluation (Self-Instruct Protocol)

Stage 1 effectively aligned the base model, transforming it from a raw text-completion engine into a conversational assistant. The untuned Base model exhibited a strong tendency to generate overly verbose completions, often hallucinating continued dialogue or formatting tags, resulting in an average response length of 211.0 tokens.

Stage 1 training aggressively pruned this behavior. The average token length dropped by nearly 80% to just 45.1 tokens per response. This increase in precision drove a massive gain in automated metrics: ROUGE-1 more than doubled from 0.2309 to 0.4712, and BERTScore F1 rose from 0.8474 to 0.9075. Consequently, in pairwise judge evaluations, Checkpoint 1 crushed the Base model, achieving an 83.9% win rate (167 wins, 10 ties out of 200 matchups).

2.3 JSON Structured Output Evaluation

Stage 2 was designed to instill strict structural discipline, and it achieved absolute mastery. At Checkpoint 0, the base model yielded a 61.0% valid parsing rate, frequently suffering from structural truncation or conversational padding.

By Checkpoint 2 (Epoch 3), the model internalized the syntax rules, achieving a 98.0% JSON validity rate. More impressively, the semantic mapping of the data within the JSON skyrocketed: Exact Match jumped from 72.9% (Checkpoint 1) to a dominant 84.7%, with a near-perfect BERTScore F1 of 0.9955 and a field-level F1 of 0.8483.

Interestingly, in pairwise JSON task evaluations judged by the LLM, Checkpoint 1 won 17 matchups while Checkpoint 2 won 15 (with a massive 68 ties). The high volume of ties indicates that both models output perfectly parsable JSON the majority of the time. The slight judge preference for Checkpoint 1, despite Checkpoint 2's vastly superior Exact Match automated metric (84.7% vs 72.9%), highlights a common artifact in AI evaluation: the LLM judge marginally favored the "chattier" output of Stage 1 over the ruthless, perfectly strict data structure of Stage 2.

2.4 Forgetting Analysis

Key Finding: Zero Catastrophic Forgetting. Sequential fine-tuning on highly rigid JSON structures did not destroy the model's conversational fluidness. In fact, Stage 2 actually won more pairwise general-instruction matchups than Stage 1.

The core objective of this research was to answer a fundamental alignment question: Does learning a rigid syntax overwrite conversational heuristics? The data answers with a resounding no.

When evaluated on 200 general Alpaca prompts, Checkpoint 2 (Stage 2) secured 72 wins compared to Checkpoint 1's 52 wins (with 76 ties), yielding a 36.0% win rate for Checkpoint 2. While automated text metrics like ROUGE-L dropped very slightly (0.4037 to 0.3869), the qualitative judge preference clearly favored Stage 2. This suggests that the structural discipline learned in the JSON dataset actually refined the model's ability to answer conversational queries concisely and accurately, effectively preventing the "mode collapse" commonly associated with sequential post-training.

3. Analysis & Discussion

3.1 Overcoming Verbosity Bias in Evaluation

A major analytical breakthrough in this project occurred during the prompt engineering of the LLM judge. Initially, the automated ROUGE and Exact Match scores contradicted the Judge's preferences on the JSON tasks. Checkpoint 2 generated far more accurate JSON payloads (84.7% Exact Match vs 72.9%), yet the generic judge preferred Checkpoint 1.

This discrepancy was isolated to Verbosity Bias. Checkpoint 1 frequently wrapped its valid JSON outputs in conversational fluff (e.g., "Certainly! Here is the extracted JSON you requested:"). The initial LLM evaluator, designed for general helpfulness, rewarded this chattiness. By engineering two highly specific evaluation prompts—one for Alpaca tasks that rewarded naturalness, and one for JSON tasks that ruthlessly penalized any non-schema text—we stripped away this bias, uncovering the true dual-capability of the Stage 2 model.

3.2 Implications for Sequential Fine-Tuning

The success of the Stage 2 model fundamentally challenges the assumption that sequential fine-tuning on narrow tasks destroys generalized capabilities. When a model is provided with high-fidelity, syntactically perfect data via imitation learning, it does not necessarily overwrite its semantic knowledge; instead, it refines its instruction-following boundaries.

For practical LLM deployment, these results imply that Epoch Tuning is critical. The ablation study revealed that Epoch 2 was the optimal stopping point, achieving 100% JSON validity and peak exact-match accuracy without over-saturating the weights, ensuring the underlying conversational agent remains fully responsive and aligned.

4. Prompt Engineering Evolution

Developing a robust end-to-end pipeline required rigorous prompt engineering on two fronts: the generation of the synthetic training data (via the teacher model) and the refinement of the automated evaluation pipeline (via the LLM-as-a-Judge).

4.1 Imitation Learning Data Generation (Teacher Prompts)

To construct the JSON Instruct dataset, we utilized the 70B teacher model to generate highly diverse, complex JSON structures. The prompts were engineered to cover five primary task categories. To ensure the student model learned robust generalization rather than memorizing a single format, each primary category was systematically divided into five distinct subcategories:

  • JSON Extraction & Schema Constrained Tasks: Taught the model how to map natural language to specific shapes (Flat Objects, Nested Objects, Arrays, Mixed, and Nullable fields).
  • JSON Repair: Prompts designed to teach syntax correction and parser compliance (fixing unquoted keys, trailing commas, truncated structures, etc.).
  • Exact-Label Classification: Prompts constraining the LLM to output specific, predefined categorical labels.
  • Tool-Calling: Prompts simulating agentic service calls (extracting parameters for flight bookings, weather APIs, etc.).

Dataset Sizing: For each of the five primary task types, we generated 70 training prompts and 20 testing prompts. This resulted in a strictly balanced and diverse dataset comprising 350 training instances and 100 evaluation instances.

4.2 LLM-as-a-Judge Prompt Splitting

As discussed in the analysis, evaluating the resulting models required splitting the LLM-as-a-Judge prompt to address inherent verbosity biases.

Alpaca Judge Prompt: Explicitly instructed to evaluate naturalness, instruction following, and conciseness, while applying CRITICAL PENALTIES for "Mode Collapse" (forcing rigid JSON structures inappropriately onto casual conversations).

JSON Judge Prompt: Transformed the judge into a ruthless syntax validator, triggering an automatic score of 1 if the model included any conversational fluff, markdown tags, or missed a schema key.

A. Final Judge Evaluation Prompts

Alpaca Judge:

You are an impartial expert evaluator of large language models. You will be given a general user prompt (like summarization, brainstorming, or QA) and two responses (A and B).

A proper response should be helpful, accurate, natural, and directly address the user's prompt in the appropriate format. 

CRITICAL PENALTIES: Severely penalize a response (score 1 for instruction_following and naturalness) if it exhibits:
- Over-Formatting / Mode Collapse: If the response forces a JSON structure, outputs Python dictionaries, or uses rigid key-value pairs for a general conversational request.
- Self-Prompting: Leaking template tags like `### Instruction:` or continuing to generate fake inputs.
- Degenerate Loops: Repeating the same phrases endlessly.

Evaluate both responses on a scale of 1-5 for: 
- instruction_following: How well does it follow the core directive AND the implied format of the request?
- correctness: Is the information factually accurate?
- clarity: Is the language clear and easy to understand?
- completeness: Does it answer all parts of the prompt?
- naturalness_and_tone: 5 means it sounds like a helpful, natural assistant. 1 means it is bizarrely robotic or forces code/JSON structures inappropriately.
- conciseness: 5 means perfectly concise; 1 means overly verbose or rambling.

You MUST output strictly in this JSON format:

JSON Judge:

You are a strict, automated JSON validation and extraction judge. You will be given a user prompt and two responses (A and B).

Your ONLY goal is to determine which response provides the most accurate, strictly formatted JSON object. 

CRITICAL PENALTIES (Automatic score of 1 for structured_output_validity):
- Conversational Fluff: If the response contains ANY text before or after the JSON (e.g., "Here is the JSON you requested:"), penalize it heavily. A perfect response is ONLY the raw JSON object or a markdown JSON block.
- Invalid JSON: Missing commas, unclosed brackets, or invalid keys.
- Schema Violation: Failing to use the exact keys requested in the prompt.

Evaluate both responses on a scale of 1-5 for: 
- structured_output_validity: 5 means perfect, parseable, isolated JSON. 1 means invalid JSON or heavy conversational text.
- schema_compliance: 5 means all requested keys/fields are present and correctly typed.
- correctness: Is the extracted data or generated content within the JSON actually correct based on the prompt?
- instruction_following: Did it perform the specific data extraction or generation task requested?
