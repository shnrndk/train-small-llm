The Alignment Tax: Sequential Fine-Tuning and Catastrophic Forgetting in LLMs

Author: Sahan Randika Kalutarage
Course/Project: LLM Post-Training & Alignment

1. Methodology

This project investigates the dynamics of sequential fine-tuning in Large Language Models (LLMs), specifically focusing on the trade-offs between acquiring new structured capabilities and retaining general conversational alignment. To explore this, we implemented a two-stage training pipeline utilizing the UTSA High-Performance Computing (HPC) cluster.

Data Sourcing and Imitation Learning

The training corpus consisted of two distinct datasets:

  • Stage 1 (General Alignment): We utilized the Alpaca dataset, a standard open-source instruction-following dataset designed to align base models to conversational and helpful assistant behaviors.
  • Stage 2 (Structured Output): We constructed a synthetic JSON Instruct dataset via Imitation Learning. We prompted a stronger teacher model to generate complex JSON responses covering five task types (extraction, summarization, etc.). This allowed the smaller student model to learn complex formatting heuristics from a state-of-the-art frontier model.

Training Design & UTSA HPC Setup

The model was trained sequentially across two stages. Checkpoint 0 represents the untuned base model. Checkpoint 1 was generated after fine-tuning on the Alpaca dataset (Stage 1). Checkpoint 2 was generated after subsequently fine-tuning Checkpoint 1 exclusively on the JSON Instruct dataset (Stage 2). Training was distributed across the UTSA HPC environment.

Evaluation Protocol

Evaluation was conducted using a hybrid approach:

  • Automated Text Metrics: ROUGE (1, 2, L) and BERTScore (using roberta-large) were used to measure semantic similarity against reference texts.
  • Automated Structural Metrics: Custom parsing scripts evaluated JSON validity, schema compliance, exact match, and field-level F1.
  • LLM-as-a-Judge: We utilized gpt-4o-mini as an impartial judge for pairwise evaluations (A/B testing) across 200 Alpaca prompts and 100 JSON prompts, scoring responses across five dimensions (Instruction Following, Correctness, Clarity, Completeness, Conciseness) on a 1-5 scale.

4.2 Alpaca Evaluation (Self-Instruct Protocol)

Stage 1 was highly successful at aligning the base model. The base model generated overly verbose, hallucinated responses (averaging 162.4 tokens). Stage 1 reduced this to a concise 45.5 tokens, drastically improving ROUGE-1 from 0.2742 to 0.4975. Consequently, Stage 1 defeated the Base model in 113 out of 200 matchups (56.5% win rate, with 10 ties).

4.3 JSON Structured Output Evaluation

Stage 2 demonstrated absolute mastery over structured generation. The validity rate reached a perfect 100%, with a BERTScore F1 of 0.9937 and a field-level F1 of 0.8197. In a direct pairwise judge evaluation, Checkpoint 2 defeated Checkpoint 1 on JSON tasks (17 wins to 12, with 71 ties, indicating both were competent, but Stage 2 was more precise).

Common Error Taxonomy: At Checkpoint 0, the model failed primarily by injecting conversational "fluff" before the JSON object or omitting closing brackets, resulting in only 31% valid parsing. By Checkpoint 2, syntax errors were entirely eliminated.

4.4 Forgetting Analysis

The Core Finding: Catastrophic forgetting occurred, but not in the traditional sense of losing factual knowledge. Instead, the model suffered from an Alignment Tax—it forgot stylistic alignment, specifically conciseness.

When evaluated on the general Alpaca prompts, Checkpoint 2 suffered a severe regression compared to Checkpoint 1, winning only 21.0% of the matchups (42 wins for Stage 2, 63 for Stage 1, 95 ties). Automated metrics confirmed this drop, with ROUGE-L falling from 0.4333 to 0.4195.

3. Analysis & Discussion

Qualitative Output Comparison & Failure Cases

The judge justifications clearly highlight the exact failure mechanism of Checkpoint 2. Because Stage 2 training exclusively featured JSON data—which requires exhaustive, complete, and highly structured data extraction—the model over-indexed on the heuristic of "exhaustiveness."

Representative Forgetting Example (Prompt: alpaca_eval_1):
When asked to summarize the US Presidential Election, Stage 1 provided a punchy summary. Stage 2 provided a much longer, bloated response. The judge ruled in favor of Stage 1, noting: "Response B [Stage 1] is more concise than Response A, omitting some details while still providing a clear summary."

Implications for Sequential Fine-Tuning

This experiment provides a textbook demonstration of why sequential fine-tuning on narrow tasks is dangerous for generalized assistants. The model did not forget how to answer questions (evidenced by the increase in Correctness and Completeness scores), but it forgot how to behave as a helpful assistant. It applied the rigid, exhaustive formatting rules of JSON data to casual conversations.

To prevent this specific type of stylistic catastrophic forgetting in future iterations, Data Composition is critical. Rather than purely sequential training, Stage 2 should utilize a mixed data curriculum—for example, 80% JSON Instruct data and 20% Alpaca data. This replay mechanism forces the model to learn new structured tasks while constantly being reminded of the concise, conversational style required for general alignment.

4. Prompt Engineering

Developing a robust pipeline required iterative prompt engineering, particularly for the LLM-as-a-Judge.

The "Illusion of Competence" Failure: Initially, the judge model suffered from "verbosity bias." It routinely awarded Checkpoint 0 (the Base model) high scores simply because it output massive walls of fluent text, completely failing to notice that the model was hallucinating contexts or leaking template tags (e.g., printing ### Instruction: mid-response).

The Fix: The judge prompt was heavily revised to include a CRITICAL PENALTIES section. We explicitly defined a "Conciseness" failure, instructing the judge to apply a score of 1 if the model exhibited "Self-Prompting," "Degenerate Loops," or unnecessary fluff. This immediately aligned the judge's scoring with human intuition, accurately penalizing the base model's word vomit and correctly identifying Stage 1's superior alignment.

Appendix: Full Prompts

A. Final Judge Evaluation Prompt

You are an impartial expert evaluator of large language models. You will be given a user prompt and two responses (A and B).

Proper response should be concise, short, accurate, and directly address the user's prompt without unnecessary fluff. 

CRITICAL PENALTIES: Severely penalize a response (score 1 for conciseness and instruction_following) if it exhibits:
- Self-Prompting: Leaking template tags like `### Instruction:` or continuing to generate fake inputs.
- Degenerate Loops: Repeating the same phrases or list structures endlessly.
- Ignoring Context: Failing to use the specific data provided in the prompt.

Evaluate both responses on a scale of 1-5 for: 
- instruction_following: How well does it follow the core directive?
- correctness: Is the information factually accurate?
- clarity: Is the language clear and easy to understand?
- completeness: Does it answer all parts of the prompt?
- conciseness: 5 means perfectly concise without missing details; 1 means overly verbose, rambling, or contains irrelevant fluff.
- structured_output_validity: Rate high if formatting is perfect JSON if requested, otherwise N/A or 5.
- hallucination_risk: 5 means NO hallucinations, 1 means high hallucination.

You MUST output strictly in this JSON format:
{
  "prompt_id": "",
  "checkpoint_a": "",
  "checkpoint_b": "",
  "response_a_scores": {"instruction_following": 0, "correctness": 0, "clarity": 0, "completeness": 0, "conciseness": 0, "structured_output_validity": 0, "hallucination_risk": 0},
  "response_b_scores": {"instruction_following": 0, "correctness": 0, "clarity": 0, "completeness": 0, "conciseness": 0, "structured_output_validity": 0, "hallucination_risk": 0},
  "winner": "",
  "justification": ""
}
